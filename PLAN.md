# План полного запуска contacts-viewer

> Цель: парсить участников WhatsApp-группы → определять, **чем каждый занимается**
> (из pushname + статуса + сообщений) → **группировать по направлениям** →
> **выгружать** (Excel + JSON) + просмотрщик. Полноценная версия, не MVP.

---

## 0. Текущее состояние (отправная точка)

Диагностика существующей выгрузки `data/group_export.json` (группа «Business Promotion», 56 чел.):

| Поле | Собрано | Проблема |
|------|---------|----------|
| `pushname` | 0/56 | выгрузка на недосинхронизированной сессии — чинится |
| `about` (статус) | ~2/56 реальных | приватность WhatsApp — потолок, не баг |
| `messages` | 0/56 | `fetchMessages` без прогрузки истории — чинится |

Вывод: текущий сбор данных сломан, `match.js` делает не то (парный матчинг вместо группировки). Переписываем сбор и второй этап.

**Жёсткий потолок:** участник, который ничего не писал, прячет статус и не в твоих контактах → доступен только номер. Это ограничение WhatsApp, не код. Финальная выборка будет меньше общего числа участников — это нормально.

---

## Архитектура (целевая)

```
scrape.js   → data/group_export.json     (сырые данные: профили + ВСЕ сообщения)
enrich.js   → data/enriched.json         (LLM: чем занимается + категория)
group.js    → data/grouped.json          (кластеризация по направлениям)
export.js   → out/people.xlsx + .json    (финальная выгрузка)
view.js     → http://localhost:5173      (просмотр, обновлён под группировку)
lib/gigachat.js                          (клиент GigaChat: токен + чат + ретраи)
config.js / .env                         (настройки и секреты)
```

Пайплайн один в один отражает твою задачу: **парсинг → определение занятия → группировка → выгрузка.**

## Этап 1 — Надёжный сбор данных (`scrape.js`, переписать)

Главный этап: без данных всё остальное бессмысленно.

1. **Дождаться полной синхронизации** после `ready` — дать сессии прогрузить контакты (проверять, что `getContacts()` непустой; ретрай с задержкой). Это вернёт `pushname`.
2. **Прогрузка истории сообщений**: перед `fetchMessages` открыть чат и **в цикле докручивать** (`loadEarlierMessages` / повторные `fetchMessages` с ростом лимита), пока не наберётся целевое число или история не кончится. Это даёт главный источник «чем занимается».
3. **Сбор по каждому участнику**: номер, `pushname`, сохранённое имя, `about`, `isAdmin`, **все его сообщения** (тело + дата + тип).
4. **Устойчивость**: try/catch по каждому контакту, ретраи, прогресс-лог, паузы между запросами (анти-бан).
5. **Статистика на выходе**: сколько участников с сообщениями / pushname / about — чтобы сразу видеть качество выгрузки.
6. **Конфиг**: лимит сообщений, задержки, headless on/off — в `config.js`.

**Результат этапа:** `data/group_export.json` с реальным текстом. Сразу смотрим статистику — если у живых участников появились сообщения, идём дальше.

---

## Этап 2 — Извлечение «чем занимается» через LLM (`enrich.js` + `lib/gigachat.js`)

1. **Клиент GigaChat** (`lib/gigachat.js`):
   - получение OAuth access-token из `Authorization key` (живёт ~30 мин → авто-рефреш);
   - решение проблемы с сертификатами «Минцифры» на Windows (корневой сертификат / настройка TLS);
   - метод `chat(messages)` с ретраями и таймаутами.
2. **Промпт на участника**: на вход pushname + about + его сообщения, на выход строгий JSON:
   ```json
   {
     "occupation": "поставки товаров из Китая",
     "category": "Логистика и ВЭД",
     "services": ["карго", "выкуп товара"],
     "location": "Баку",
     "confidence": 0.0-1.0,
     "summary": "короткое описание"
   }
   ```
3. **Очистка ответа**: снятие ```-обёрток перед `JSON.parse`, валидация схемы.
4. **Rate limiting + батчи**: ограничение параллелизма (свой лимитер, без `p-limit`), паузы — не упереться в лимиты API.
5. **Кэш**: результат по `id` сохраняется → повторный прогон не дёргает API заново и не жжёт лимит.
6. **Фолбэк**: у кого нет текста → `category: "Неизвестно / молчун"`, `confidence: 0`. Не выкидываем, но помечаем.

**Результат этапа:** `data/enriched.json` — у каждого размечено занятие и категория.

---

## Этап 3 — Группировка по направлениям (`group.js`)

1. Собрать всех по `category`.
2. **Нормализация категорий** вторым проходом LLM: привести разнобой («стройка», «ремонт квартир», «отделка») к единому списку направлений, убрать дубли-синонимы.
3. Внутри группы — сортировка по `confidence` (сначала уверенные).
4. Сводка: сколько человек в каждом направлении, сколько «молчунов».

**Результат этапа:** `data/grouped.json` — чистые группы «направление → список людей».

---

## Этап 4 — Выгрузка (`export.js`)

1. **Excel (`out/people.xlsx`)** через `exceljs`:
   - лист **«Все»**: имя, номер, занятие, категория, услуги, локация, источник (сообщение/статус/имя), уверенность;
   - по листу **на каждое направление** (или группировка/фильтры на одном листе);
   - лист **«Сводка»**: счётчики по категориям.
2. **JSON** (`out/people.json`) — машинная версия для дальнейшего использования.
3. Номера в безопасном формате (текст, чтобы Excel не съел `+`).

**Результат этапа:** готовый файл, который можно открыть/переслать.

## Этап 6 — Эксплуатация и надёжность

- **Секреты**: `Authorization key` GigaChat — в `.env`, добавить в `.gitignore` (вместе с `.wwebjs_auth/`, `data/`, `out/`).
- **Логирование**: понятные шаги и итоговая статистика на каждом этапе.
- **Анти-бан**: задержки в `scrape.js`, не гонять выгрузку часто, предупреждение про ToS WhatsApp (оно уже есть в README).
- **Resume**: кэш на этапе LLM + отдельные файлы между этапами → любой шаг перезапускается независимо.
- **Конфиг** (`config.js`): лимиты сообщений, параллелизм LLM, модель GigaChat, целевые категории.

## Этап 7 — Запуск (runbook под Windows / PowerShell)

```powershell
# зависимости (qrcode + опционально exceljs)
npm install

# 1. собрать данные (QR при первом запуске)
npm run scrape

# 2. разметить занятия через GigaChat (нужен ключ в .env)
npm run enrich

# 3. сгруппировать
npm run group

# 4. выгрузить в Excel + JSON
npm run export

# 5. посмотреть в браузере
npm run view   # http://localhost:5173
```

`.env`:
```
GIGACHAT_AUTH_KEY=...      # Authorization key из кабинета Sber
GIGACHAT_SCOPE=GIGACHAT_API_PERS
GIGACHAT_MODEL=GigaChat-Pro
```

## Порядок работ

1. **Этап 1** — переписать `scrape.js`, перевыгрузить, посмотреть честную статистику. ← **старт**
2. Если данные появились → **Этап 2** (клиент GigaChat + enrich).
3. **Этап 3–4** — группировка и выгрузка.
4. **Этап 5–6** — просмотрщик и обвязка.

Главная развилка — после Этапа 1: реальное количество текста определит, насколько богатой будет выгрузка. Всё остальное по коду делается уверенно.
